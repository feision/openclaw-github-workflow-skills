## 📋 事件概述

尝试修复远程仓库中存在的嵌套目录问题（`openclaw-github-workflow-skills/` 子目录），过程中**错误地重写了 Git 历史**，导致：

- 原始提交链暂时丢失
- 强制推送覆盖了远程 `main` 分支
- 用户对版本历史完整性产生担忧

## 🔄 事件时间线

### 1. 问题发现
- 远程仓库文件结构异常：所有教程文件出现在 `openclaw-github-workflow-skills/` 子目录下
- 本地克隆同样存在此问题
- 从 GitHub 下载的官方 zip 包也包含嵌套结构

### 2. 错误的修复尝试

| 步骤 | 操作 | 问题 |
|------|------|------|
| 1 | 尝试 `git push --force` 覆盖 | 未清除所有 tree 对象，嵌套依然存在 |
| 2 | 创建 `temp_branch` 并 `git add -A` | 误将父目录文件纳入仓库，问题恶化 |
| 3 | 删除嵌套目录后 `--orphan` 重建 | 历史完全断裂，提交只剩一个 |
| 4 | 强制推送新仓库到远程 | **原始历史被覆盖**，这是最严重的错误 |

### 3. 问题识别与恢复

- 检查 `git ls-remote origin`，发现远程仍有完整的原始引用
- 发现 `origin/main` 实际上指向 `6d5c78e`（"Delete index.html"）
- 执行 `git reset --hard origin/main`
- **历史完整恢复**，嵌套问题已在早先提交中修复

## 🎯 根本原因分析

1. **对 Git 对象模型理解不足**：
   - Git 的 tree 对象不可变，`--force` 只能添加新对象，无法删除被引用的旧对象
   - 嵌套目录在早期提交中已引入，需要 `git filter-branch` 或 `git rebase` 修正，而非暴力重建

2. **缺乏备份与验证**：
   - 未先 `git fetch --all` 查看所有引用
   - 未确认哪些分支/PR 引用可能影响操作
   - 没有创建备份分支（如 `backup/pre-incident`）

3. **操作顺序错误**：
   - 应先 `git fetch` 然后检查 `origin/HEAD` 和 `origin/main`
   - 应使用 `git replace` 或 `git filter-repo` 清理历史，而不是删除 `.git`

4. **沟通不足**：
   - 未及时告知用户风险
   - 在多次尝试失败后未请求暂停或寻求更优方案

## 📚 正确的修复流程（本应如此）

```bash
# 1. 获取所有引用，评估影响
git fetch --all --prune
git ls-remote origin

# 2. 创建备份分支（防止误操作）
git branch backup/pre-incident origin/main

# 3. 找到最早引入嵌套目录的提交
git log --oneline --diff-filter=A -- "openclaw-github-workflow-skills/"

# 4. 使用 filter-branch 或 filter-repo 清理
# 示例：移除整个目录
git filter-branch --tree-filter 'rm -rf openclaw-github-workflow-skills' --prune-empty HEAD

# 或使用 filter-repo（推荐）
git filter-repo --path openclaw-github-workflow-skills --invert-paths

# 5. 验证本地仓库结构
ls -la | grep openclaw

# 6. 推送（如果无外部引用，可用 --force-with-lease）
git push origin main --force-with-lease

# 7. 清理旧引用（如需要）
git push origin --delete clean-main Openclaw
```

## 🛡️ 未来预防措施

1. **操作前必做**：
   - `git status` 和 `git log --oneline -5` 确认当前状态
   - `git ls-remote origin` 查看远程所有分支
   - 创建备份分支：`git branch backup/YYYY-MM-DD-HHMM`

2. **重写历史时**：
   - 使用 `--force-with-lease` 而非 `--force`
   - 通知所有协作者暂停推送
   - 检查是否有 open PRs 可能受影响

3. **问题排查优先顺序**：
   - 先 `git fetch --all` 确保本地信息最新
   - 用 `git fsck` 检查仓库完整性
   - 考虑是否可用 `git revert` 而非重写

4. **文档记录**：
   - 任何破坏性操作前，先在 MEMORY.md 或 incident 文件中记录计划
   - 操作后验证结果并总结
